Q1. What are you using for implementing WebSockets?

We use Socket.io on Node.js with the Socket.io Redis adapter (Upstash Redis).

When a user connects, their JWT is verified before the handshake completes. The server then joins them to Socket.io "rooms" for every conversation they belong to. Sending a message to a room instantly delivers it to all online members. The Redis adapter means all this works transparently across multiple server instances — adding a new server requires zero code changes.

Raw WebSockets are the browser primitive. Socket.io is the production wrapper. The question is not raw vs Socket.io but rather: spend three weeks building reconnection, room management, and delivery acknowledgements from scratch, or use a battle-tested library and ship features instead? Socket.io is used in production by Trello, Notion, and many large-scale real-time apps.

Why the Redis adapter is the most important decision here

Without Redis: User A is on Server 1, User B is on Server 2. User A sends a message. Server 1 broadcasts to its local room. User B never receives it because they are on a different server.

With the Redis adapter, each server publishes events to a shared Redis channel. Every other server subscribes and delivers to its local connections. Adding a new server instance requires zero code changes.

// Works transparently across ALL server instances

Path to 1 million users

Step 1 (10K to 100K): Deploy 5 to 10 Node instances behind a load balancer. Redis adapter is already in place, zero code changes needed.

Step 2 (100K to 500K): Split into dedicated Redis clusters per concern: one for Socket.io, one for BullMQ, one for presence. Configuration change only.

Step 3 (500K to 1M+): Deploy servers in multiple regions with GeoDNS routing. This is how Discord runs across 20+ global regions.

Q2. How would you support other message types like audio, documents, etc?

Binary files never pass through our server. We use the presigned URL pattern: the client asks our server for a signed upload URL, uploads the file directly to Cloudflare R2 (bypassing our server entirely), then sends a WebSocket message containing just the CDN URL and metadata.

This means our server never touches binary data. It just validates, signs, and steps aside. R2 handles the actual transfer. Adding a new message type (voice notes, location, polls) only requires a new schema shape and a UI component. The transport layer never changes.

Why not POST the file through our server?

Memory: A 100MB video held in memory consumes 100MB of RAM per concurrent upload. At 1,000 simultaneous uploads that is 100GB of RAM, which is impossible on any practical server.

Bandwidth: Every byte travels through our server twice — in on upload, out on re-upload to storage. Double the cost, double the latency.

Blocking: Node.js is single-threaded. Streaming large files blocks the event loop and degrades performance for every other connected user.

Client-side compression and Blurhash

Images over 1MB are resized to 1920px max and re-encoded as WebP at 85% quality in the browser before upload. An 8MB phone photo becomes around 600KB. At 10,000 uploads, storage costs drop by 92%.

A Blurhash (30-byte blurred preview string) is generated and sent with the message. The receiver renders the blurred placeholder instantly with no network request, then loads the full image in the background. This is the same technique used by Medium, Facebook, and Notion.

Path to 1 million users

Cloudflare R2 has no egress fees, a significant cost advantage over AWS S3. At 1M users, files over 5MB should use multipart upload: the client splits the file into 5MB chunks and uploads in parallel. A 100MB video drops from 60 seconds to 15 seconds.

Q3. How do we check if someone is typing?

Typing indicators are purely ephemeral and never touch a database. The client emits a typing:start event on the first keystroke and a typing:stop event 1.5 seconds after the last keystroke (debounced). The server broadcasts these to all other room members.

A 5-second server-side safety timer auto-clears stale state if a client crashes without sending typing:stop. Typing state lives only in an in-memory Map on the server. If the server restarts, it disappears — which is exactly the correct behaviour.

Why no database, ever

Typing state has no business value after the fact. Writing it to a database means thousands of writes per second that are never read again. Even Redis is overkill for a single-server deployment. We use an in-memory Map because it is the right tool for transient state.

const typingTimeouts = new Map();

socket.on('typing:start', (payload) => {
  socket.to(`conversation:${payload.conversationId}`)
    .emit('typing:start', { ...payload, userId });

  const key = `${userId}:${payload.conversationId}`;
  clearTimeout(typingTimeouts.get(key));
  typingTimeouts.set(key, setTimeout(() => {
    socket.to(`conversation:${payload.conversationId}`)
      .emit('typing:stop', { userId });
    typingTimeouts.delete(key);
  }, 5000));
});

socket.on('disconnect', () => {
  typingTimeouts.forEach((timeout, key) => {
    if (key.startsWith(userId)) {
      clearTimeout(timeout);
      typingTimeouts.delete(key);
    }
  });
});

Why we debounce on the client, not the server

A user typing a 500-character message sends exactly 2 events: typing:start and typing:stop. Without client debouncing, every keystroke emits an event — 500 events for the same result. The general principle is to filter noise as close to the source as possible.

Path to 1 million users

The in-memory approach has one limitation on multi-server deployments: a typing timeout on Server 1 does not clean up if the disconnect happens on Server 2. The fix takes about 10 minutes: store typing state in Redis with a 5-second TTL. The key expires automatically, no cleanup code needed.

await redis.setex(`typing:${userId}:${conversationId}`, 5, '1');
// Redis expires the key automatically after 5 seconds
Q4. How can we separate Admin, Read, or Write members in a group?

Every group member has a role (ADMIN, WRITE, or READ) stored in a PostgreSQL junction table. ADMIN can manage members. WRITE can send messages. READ can only receive.

Enforcement happens in two independent layers: an Express middleware on REST routes, and an explicit role check inside every WebSocket event handler. A READ member still joins the Socket.io room (so they receive messages) but their send events are rejected immediately with a permission denied acknowledgement.

Data model

CREATE TABLE conversation_members (
  conversation_id UUID REFERENCES conversations(id) ON DELETE CASCADE,
  user_id         UUID REFERENCES users(id) ON DELETE CASCADE,
  role            member_role NOT NULL DEFAULT 'WRITE',
  UNIQUE(conversation_id, user_id)
);

CREATE TYPE member_role AS ENUM ('ADMIN', 'WRITE', 'READ');

Two enforcement layers, not one

HTTP middleware only protects REST routes. A malicious client can open a WebSocket and emit events directly, bypassing HTTP entirely. So we check the role again inside every relevant event handler.

// Layer 1: HTTP middleware
const requireRole = (minimumRole) => async (req, res, next) => {
  const member = await prisma.conversationMember.findUnique({...});
  const ROLE_RANK = { ADMIN: 3, WRITE: 2, READ: 1 };
  if (ROLE_RANK[member.role] < ROLE_RANK[minimumRole]) {
    throw new ForbiddenError(`Requires ${minimumRole} role`);
  }
  next();
};

// Layer 2: WebSocket handler
socket.on('message:send', async (payload, ack) => {
  const member = await ConvoRepo.findMember(payload.conversationId, userId);
  if (member.role === 'READ') {
    return ack({ status: 'error', code: 'PERMISSION_DENIED' });
  }
  // Only WRITE and ADMIN reach here
  const message = await MessageService.send(userId, payload);
  io.to(`conversation:${payload.conversationId}`).emit('message:new', message);
  ack({ status: 'ok', seq: message.seq });
});

Why PostgreSQL for roles and not MongoDB

Roles are relational data. The UNIQUE constraint ensures a user can never hold two roles in the same group — the database enforces this, not application code. Role changes must also be immediately consistent: if an admin demotes someone to READ, the very next WebSocket event from that user must reflect it. PostgreSQL ACID guarantees this. Messages are high-volume append-only documents. MongoDB is better suited for that. Using the right database for the right problem is what makes systems maintainable at scale.

Path to 1 million users

Step 1 (10K to 500K): Cache roles in Redis for 5 minutes. Invalidate on role change. Reduces Postgres load by 90%.

let role = await redis.get(`role:${conversationId}:${userId}`);
if (!role) {
  role = (await prisma.conversationMember.findUnique(...))?.role;
  await redis.setex(`role:${conversationId}:${userId}`, 300, role);
}

Step 2 (500K to 1M): On connect, load all roles once and store on the socket object. Role checks become in-memory Map lookups — zero network calls per event.

// Zero DB calls on every message

1. User logs in and receives a JWT access token (15 min expiry)

3. Server verifies the JWT before the handshake completes

5. User is joined to a Socket.io "room" for each conversation
   Room naming pattern: "conversation:{uuid}"
6. Any message emitted to a room reaches all members instantly:
   io.to(`conversation:${id}`).emit('message:new', message)

The naive approach sends the file to our server which then uploads to storage. This has three problems that make it unworkable at scale.

Memory: A 100MB video held in memory consumes 100MB of RAM per concurrent upload. At 1,000 concurrent uploads that is 100GB of RAM, which is simply impossible on any practical server.

Blocking: Node.js runs on a single thread. Streaming large files blocks the event loop and degrades performance for every other user connected to that server.

The presigned URL pattern eliminates all three. Our server does a few milliseconds of validation and URL signing, then steps aside. R2 handles the actual transfer.

Client-side compression

Before the upload even starts, the client compresses images that are over 1MB. The image is resized to a maximum of 1920px on the longest side and re-encoded as WebP at 85% quality. A typical 8MB phone photo becomes around 600KB. This happens entirely in the browser at zero server cost.

At 10,000 users uploading 8MB photos, the uncompressed total is 80GB. With compression it is 6GB. Storage costs drop by 92% and CDN delivery is faster because smaller files transfer quicker to the recipient.

Blurhash previews

When an image is uploaded, the client generates a Blurhash: a 30-byte string encoding a blurred preview of the image. This string is sent with the message metadata. The receiver renders the blurred placeholder immediately from those 30 bytes, with no additional network request, then loads the full image in the background. The user perceives the image as instant even on a slow connection. This is the same technique used by Medium, Facebook, and Notion.

Q4. How can we separate Admin, Read, or Write members in a group?

We use Role-Based Access Control (RBAC) enforced at both the HTTP middleware layer and the WebSocket event handler layer.

Data model in PostgreSQL

CREATE TABLE conversation_members (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  conversation_id UUID REFERENCES conversations(id) ON DELETE CASCADE,
  user_id         UUID REFERENCES users(id) ON DELETE CASCADE,
  role            member_role NOT NULL DEFAULT 'WRITE',
  joined_at       TIMESTAMP DEFAULT NOW(),
  last_read_at    TIMESTAMP DEFAULT NOW(),

  UNIQUE(conversation_id, user_id)
);

CREATE TYPE member_role AS ENUM ('ADMIN', 'WRITE', 'READ');

Role hierarchy

ADMIN  (rank 3)
  Can do everything WRITE can, plus:
  Add or remove members, change roles, edit group name and avatar

WRITE  (rank 2)
  Can do everything READ can, plus:
  Send text and media, edit own messages, delete own messages

READ   (rank 1)
  Receive messages, see members, see typing indicators
  Cannot send anything. Attempts are rejected silently.

Enforcement happens in two places, not one

A single enforcement point is a single point of failure. We check roles in two independent layers so that bypassing one does not help an attacker.

Layer 1: HTTP middleware

const requireRole = (minimumRole: MemberRole) =>
  async (req, res, next) => {
    const member = await prisma.conversationMember.findUnique({
      where: { conversationId_userId: {
        conversationId: req.params.conversationId,
        userId: req.user.id
      }}
    });

    if (!member) throw new NotFoundError('Conversation');

    const ROLE_RANK = { ADMIN: 3, WRITE: 2, READ: 1 };
    if (ROLE_RANK[member.role] < ROLE_RANK[minimumRole]) {
      throw new ForbiddenError(`Requires ${minimumRole} role`);
    }

    req.memberRole = member.role;
    next();
  };

router.post('/:id/members',        authMiddleware, requireRole('ADMIN'), addMember);
router.delete('/:id/members/:uid', authMiddleware, requireRole('ADMIN'), removeMember);
router.post('/messages',           authMiddleware, requireRole('WRITE'), sendMessage);

Layer 2: WebSocket handler

HTTP middleware only protects REST routes. A malicious client can open a WebSocket connection and emit events directly, bypassing HTTP entirely. So we check the role again inside every relevant event handler.

socket.on('message:send', async (payload, ack) => {
  const member = await ConvoRepo.findMember(payload.conversationId, userId);

  if (!member) {
    return ack({ status: 'error', code: 'NOT_A_MEMBER' });
  }

  if (member.role === 'READ') {
    return ack({ status: 'error', code: 'PERMISSION_DENIED' });
  }

  // Only WRITE and ADMIN reach this point
  const message = await MessageService.send(userId, payload);
  io.to(`conversation:${payload.conversationId}`).emit('message:new', message);
  ack({ status: 'ok', seq: message.seq });
});

READ members still join the Socket.io room so they receive messages. They simply cannot emit anything that has an effect.

Why PostgreSQL for roles and not MongoDB

Roles are relational data. A role belongs to a user AND a conversation simultaneously. The UNIQUE constraint on the table ensures a user can never accidentally hold two roles in the same group. That guarantee comes from the database, not from application code. Role changes must also be immediately consistent: if an admin demotes someone to READ, that change must be enforced on the very next WebSocket event from that user. PostgreSQL's ACID guarantees this. Messages, by contrast, are high-volume append-only documents with no strict consistency requirements. MongoDB is better suited for that access pattern. Using the right database for the right problem is what makes systems maintainable at scale.

Path to 1 million users

Step 1 (10K to 500K users): Cache roles in Redis. Currently we make one PostgreSQL query per WebSocket event to check the member's role. At 100K messages per second, that is 100K Postgres queries per second, which is too many. Caching the role in Redis for 5 minutes and invalidating on role changes reduces database load by around 90%.

const cacheKey = `role:${conversationId}:${userId}`;
let role = await redis.get(cacheKey);

if (!role) {
  const member = await prisma.conversationMember.findUnique(...);
  role = member?.role;
  await redis.setex(cacheKey, 300, role);
}

Step 2 (500K to 1M users): Attach roles to the socket on connection. On connect, load all conversation roles for that user once and store them on the socket object. Role checks then become in-memory Map lookups with zero network calls. When an admin changes a role, emit a role:updated event to the affected user's socket to refresh their local map.

// On connect, load all roles once
socket.data.roles = new Map(
  memberships.map(m => [m.conversationId, m.role])
);

// On every message:send, zero network calls
const role = socket.data.roles.get(payload.conversationId);
